Why 2× the accesses?
One vlse32 with stride 76 bytes touches 32 elements — each on a separate 64-byte cache line. Scalar accesses the same data sequentially; the prefetcher hides latency. RVV forces the memory subsystem to service 32 simultaneous cache-line requests per instruction. More accesses, more bus traffic, more dirty evictions.

RVV has better L2 reuse but worse L1D bandwidth
The strided loads bring in more cache lines per instruction; those lines get reused within L2, so the L2 miss rate drops from 54.6% to 22.9%. When a miss does reach L2, it resolves faster — hence the lower avg L1D miss latency for RVV. The problem is not L2 capacity: it is L1D bandwidth saturation. Strided loads are displacing the scalar working set from L1D, forcing dirty evictions (3.4× more writebacks) and re-fetches.
Core problem — L1D access doubling
RVV generates 104k L1D accesses vs scalar's 56k — nearly 2×. This is vlse32 working exactly as designed but against the wrong memory layout. Each vlse32 instruction touches 32 elements separated by 76 bytes, fetching across 32 different cache lines in a single instruction. The scalar loop accesses the same data but sequentially, allowing the prefetcher to hide latency.
Counterintuitive — L2 miss rate is better under RVV
Scalar L2 data miss rate is 54.6% vs RVV's 22.9% — RVV actually has better L2 reuse. This makes sense: the strided loads bring in more cache lines per instruction, and those lines get reused within L2. The problem is not L2 capacity — it is L1D bandwidth saturation.
Writeback explosion confirms cache thrashing
RVV generates 6,395 writebacks vs scalar's 1,906 — 3.4× more dirty evictions. The strided loads are displacing clean working-set data from L1D, forcing it to be written back to L2 and then re-fetched on subsequent iterations.
Lower avg miss latency for RVV is not a win
RVV's L1D avg miss latency is 21k ticks vs scalar's 39k. This looks like an improvement but is an artefact: misses that reach L2 are finding data there (22.9% L2 miss rate vs 54.6%), so the L2→L1D fill is faster on average. The absolute miss count is still 2.4× higher.

Root cause: layout mismatch. conv_x stride exceeds one cache line. Fix: transpose so d_conv is the contiguous axis → vlse32 becomes vle32 → unit-stride → prefetcher works → L1D pressure collapses.
